This guide walks through day-to-day operation of a self-hosted SwissKnife instance: signing in, checking your balance, receiving and sending payments over Lightning and on-chain Bitcoin, and automating everything with scoped API keys.

Every operation described here is available both from the **Dashboard UI** and from the **REST API**. The full, always-up-to-date endpoint reference (request/response schemas, required permissions, and an interactive playground) lives at [api.numeraire.tech/docs](https://api.numeraire.tech/docs). When self-hosting, the same API is served by your own instance (default port `3000` for the all-in-one image), so substitute your instance URL for the examples below.

:::tip
The `/v1/me` endpoints are the main access point for an authenticated account. Account settings and API keys are account-scoped, while balances, invoices, payments, contacts, and Bitcoin addresses require an explicit wallet path: `/v1/me/wallets/{wallet_id}/...`.
:::

## Signing in

SwissKnife self-hosted is single-tenant by default: on first launch you create one initial **Admin account**. There is no static admin password in the configuration file; it is set once through the sign-up flow and stored hashed in the database.

### First launch (initial setup)

When the dashboard starts, it checks setup status via [`GET /v1/system/setup`](https://api.numeraire.tech/docs#tag/system/GET/v1/system/setup), which returns:

- `welcome_complete` — whether the welcome flow has been completed.
- `sign_up_complete` — whether the Admin account has been created.

Complete the welcome step (the dashboard calls [`POST /v1/system/mark-welcome-complete`](https://api.numeraire.tech/docs#tag/system/POST/v1/system/mark-welcome-complete)), then create the Admin account by setting a password. Sign-up only collects a password (entered and confirmed); there are no name, email, or profile fields.

```bash
# Create the initial Admin account (JWT auth provider only)
curl -X POST https://your-instance:3000/v1/auth/sign-up \
  -H 'Content-Type: application/json' \
  -d '{ "password": "<your-admin-password>" }'
# => { "token": "<jwt>" }
```

:::note
Sign-up is only available when SwissKnife is configured with the `JWT` auth provider. If you use OAuth2 / OIDC, the Admin identity and login are handled by your external OpenID provider via redirect, and you do not create a password here. See the [Authentication](../configuration/authentication.mdx) configuration page.
:::

### Subsequent logins

Authenticate with the password you set to obtain a JWT:

```bash
curl -X POST https://your-instance:3000/v1/auth/sign-in \
  -H 'Content-Type: application/json' \
  -d '{ "password": "<your-admin-password>" }'
# => { "token": "<jwt>" }
```

Send the returned token as a Bearer credential on subsequent requests:

```bash
curl https://your-instance:3000/v1/me \
  -H 'Authorization: Bearer <jwt>'
```

The initial local-JWT account is granted all administrative permissions. For programmatic access
with independently revocable credentials and no unnecessary admin scopes, use
[API keys](#managing-api-keys) instead.

## Checking your balance

Fetch your account, select one of its wallets, then request the balance for that wallet:

- [`GET /v1/me`](https://api.numeraire.tech/docs#tag/me/GET/v1/me) — your account profile and lightweight wallet list.
- [`PUT /v1/me`](https://api.numeraire.tech/docs#tag/me/PUT/v1/me) — update your account display name.
- [`GET /v1/me/wallets`](https://api.numeraire.tech/docs#tag/me/GET/v1/me/wallets) — wallets owned by your account.
- [`GET /v1/me/wallets/{wallet_id}/balance`](https://api.numeraire.tech/docs#tag/me/GET/v1/me/wallets/{wallet_id}/balance) — the selected wallet's balance breakdown.

```bash
curl https://your-instance:3000/v1/me/wallets \
  -H 'Authorization: Bearer <jwt>'

curl https://your-instance:3000/v1/me/wallets/<wallet-id>/balance \
  -H 'Authorization: Bearer <jwt>'
```

Wallets embedded in an account include asset metadata and balances, but omit payments, invoices,
addresses, contacts, and other related resources. Fetch the selected wallet or one of its child
routes by ID when that data is needed.

Native BTC balances are reported in millisatoshis (`msat`) with these fields:

| Field            | Meaning                                          |
| ---------------- | ------------------------------------------------ |
| `available_msat` | Amount available to spend.                       |
| `reserved_msat`  | Amount reserved for pending outgoing payments.   |
| `received_msat`  | Total amount received.                           |
| `sent_msat`      | Total amount sent (settled outgoing payments).   |
| `fees_paid_msat` | Total routing/network fees paid.                 |

Account-scoped `/v1/me` routes require authentication and enforce wallet ownership; they do not
consume administrative permission scopes.

The dashboard loads the full account wallet list into its account context. The wallet selector in
the header controls every wallet-scoped page and persists `active_wallet_id` in your account
preferences; changing wallets does not infer or silently fall back to the first wallet.

## Receiving

SwissKnife can receive funds over Lightning (invoices, your Lightning Address, LNURL-pay, and Nostr Zaps) and on-chain Bitcoin.

### Lightning invoices

Generate a BOLT11 invoice for a specific amount:

- [`POST /v1/me/wallets/{wallet_id}/invoices`](https://api.numeraire.tech/docs#tag/me/POST/v1/me/wallets/{wallet_id}/invoices) — generate an invoice for the selected wallet.
- [`GET /v1/me/wallets/{wallet_id}/invoices`](https://api.numeraire.tech/docs#tag/me/GET/v1/me/wallets/{wallet_id}/invoices) and [`GET /v1/me/wallets/{wallet_id}/invoices/{id}`](https://api.numeraire.tech/docs#tag/me/GET/v1/me/wallets/{wallet_id}/invoices/{id}) — list and look up invoices for that wallet.

```bash
curl -X POST https://your-instance:3000/v1/me/wallets/<wallet-id>/invoices \
  -H 'Authorization: Bearer <jwt>' \
  -H 'Content-Type: application/json' \
  -d '{
    "amount_msat": 100000,
    "description": "Coffee",
    "expiry": 3600
  }'
```

Only `amount_msat` is required; `description` and `expiry` (seconds) are optional. The
account-scoped routes authorize access through the authenticated account and wallet path.

### Your Lightning Address

A Lightning Address (`username@your.domain`) lets anyone pay you using a human-readable identifier, the same way email works. The domain is configured per instance (see [Configuration](../configuration/index.mdx)).

- [`POST /v1/me/lightning-address`](https://api.numeraire.tech/docs#tag/me/POST/v1/me/lightning-address) — register your address (`username`).
- [`GET /v1/me/lightning-address`](https://api.numeraire.tech/docs#tag/me/GET/v1/me/lightning-address) — fetch your registered address.
- [`PUT /v1/me/lightning-address`](https://api.numeraire.tech/docs#tag/me/PUT/v1/me/lightning-address) — update it (toggle `active`, Nostr settings).

```bash
curl -X POST https://your-instance:3000/v1/me/lightning-address \
  -H 'Authorization: Bearer <jwt>' \
  -H 'Content-Type: application/json' \
  -d '{
    "username": "alice",
    "allows_nostr": true,
    "nostr_pubkey": "<hex-pubkey>"
  }'
```

Lightning Addresses are account-level handles. SwissKnife routes incoming invoices to the
account's native-BTC wallet for the node's configured network; callers cannot select an
incompatible network or a non-BTC asset. Account ownership authorizes the `/v1/me` registration,
update, read, and delete routes.

### LNURL-pay

Each registered Lightning Address is served over LNURL-pay at well-known, unauthenticated endpoints that any LNURL-capable wallet can resolve:

- [`GET /.well-known/lnurlp/{username}`](https://api.numeraire.tech/docs#tag/lnurl/GET/.well-known/lnurlp/{username}) — returns the LNURL `payRequest` for the address.
- [`GET /lnurlp/{username}/callback`](https://api.numeraire.tech/docs#tag/lnurl/GET/lnurlp/{username}/callback) — returns an invoice for the requested amount.

You do not call these yourself; they are what a sender's wallet hits when paying `username@your.domain`.

### Nostr (NIP-05 and Zaps)

When a Lightning Address is registered with `allows_nostr: true` and a `nostr_pubkey`, SwissKnife also serves NIP-05 verification and enables Lightning Zaps for that identifier:

- [`GET /.well-known/nostr.json`](https://api.numeraire.tech/docs#tag/nostr/GET/.well-known/nostr.json) — maps the username to its Nostr public key (hex), satisfying [NIP-05](https://github.com/nostr-protocol/nips/blob/master/05.md).

### On-chain Bitcoin address

For self-hosted CLN and LND setups, SwissKnife includes an integrated on-chain wallet. Generate a deposit address to receive on-chain Bitcoin:

- [`POST /v1/me/wallets/{wallet_id}/bitcoin/addresses`](https://api.numeraire.tech/docs#tag/me/POST/v1/me/wallets/{wallet_id}/bitcoin/addresses) — get a deposit address for the selected wallet.

```bash
curl -X POST https://your-instance:3000/v1/me/wallets/<wallet-id>/bitcoin/addresses \
  -H 'Authorization: Bearer <jwt>' \
  -H 'Content-Type: application/json' \
  -d '{}'
```

The returned address stays the same until it is used on-chain, after which a new one is generated. You can optionally request a specific script `type` (`p2pkh`, `p2sh`, `p2wpkh`, or `p2tr`); when omitted, the instance default (`bitcoin_address_type`, defaults to `p2wpkh`) is used. The account-scoped route enforces wallet ownership. See the [on-chain wallet](#on-chain-bitcoin-wallet) section below for managing addresses.

## Sending

A single endpoint handles all outgoing payments. The recipient `input` can be a BOLT11 invoice, an LNURL, or a Lightning Address; payments to another wallet on the same instance are settled instantly through the internal ledger.

- [`POST /v1/me/wallets/{wallet_id}/payments`](https://api.numeraire.tech/docs#tag/me/POST/v1/me/wallets/{wallet_id}/payments) — send a payment from the selected wallet.
- [`GET /v1/me/wallets/{wallet_id}/payments`](https://api.numeraire.tech/docs#tag/me/GET/v1/me/wallets/{wallet_id}/payments) and [`GET /v1/me/wallets/{wallet_id}/payments/{id}`](https://api.numeraire.tech/docs#tag/me/GET/v1/me/wallets/{wallet_id}/payments/{id}) — list and look up payments for that wallet.

```bash
# Pay a BOLT11 invoice
curl -X POST https://your-instance:3000/v1/me/wallets/<wallet-id>/payments \
  -H 'Authorization: Bearer <jwt>' \
  -H 'Content-Type: application/json' \
  -d '{ "input": "lnbc1..." }'

# Pay a Lightning Address (amount required; the input has none)
curl -X POST https://your-instance:3000/v1/me/wallets/<wallet-id>/payments \
  -H 'Authorization: Bearer <jwt>' \
  -H 'Content-Type: application/json' \
  -d '{ "input": "bob@example.com", "amount_msat": 100000, "comment": "Thanks!" }'
```

Notes on the request body:

- `input` is required. For a BOLT11 invoice that already encodes an amount, omit `amount_msat`; for an LNURL or Lightning Address, supply `amount_msat`.
- `comment` is optional and is shown to the recipient for LNURL payments.

The account-scoped send and list routes enforce ownership of the wallet in the path.

:::tip
Admins can send on behalf of any wallet using the non-`/me` endpoint [`POST /v1/payments`](https://api.numeraire.tech/docs#tag/payments/POST/v1/payments) with a `wallet_id`. There is no default wallet fallback; the wallet selector is required.
:::

## On-chain Bitcoin wallet

The on-chain wallet (available for self-hosted CLN and LND) supports address management, wallet synchronization, and transaction preparation before broadcast.

### Generating and managing addresses

- [`POST /v1/me/wallets/{wallet_id}/bitcoin/addresses`](https://api.numeraire.tech/docs#tag/me/POST/v1/me/wallets/{wallet_id}/bitcoin/addresses) — generate/return a deposit address for the selected wallet (see [On-chain Bitcoin address](#on-chain-bitcoin-address) above).
- [`GET /v1/bitcoin/addresses`](https://api.numeraire.tech/docs#tag/bitcoin-addresses/GET/v1/bitcoin/addresses) — list Bitcoin addresses (with filtering and pagination).
- [`GET /v1/bitcoin/addresses/{id}`](https://api.numeraire.tech/docs#tag/bitcoin-addresses/GET/v1/bitcoin/addresses/{id}) — look up a single address.
- [`DELETE /v1/bitcoin/addresses`](https://api.numeraire.tech/docs#tag/bitcoin-addresses/DELETE/v1/bitcoin/addresses) and [`DELETE /v1/bitcoin/addresses/{id}`](https://api.numeraire.tech/docs#tag/bitcoin-addresses/DELETE/v1/bitcoin/addresses/{id}) — delete addresses.

Each address record reports its `address`, `address_type`, and a `used` flag indicating whether it has already received funds on-chain. The non-`/me` administrative list/get routes require `read:btc_address`, and administrative generation/deletion requires `write:btc_address`.

### Wallet sync and transaction preparation

SwissKnife synchronizes on-chain wallet state with your node so deposits are detected and balances stay current. Outgoing on-chain spends are **prepared before broadcast**, giving you a safe path to construct and review a transaction. On-chain spends are sent through the same payment flow as Lightning: pass an on-chain destination to [`POST /v1/me/wallets/{wallet_id}/payments`](https://api.numeraire.tech/docs#tag/me/POST/v1/me/wallets/{wallet_id}/payments).

:::note
The on-chain wallet requires a self-hosted Core Lightning or LND provider and the `bitcoin_address_type` setting. See the [Lightning providers](../configuration/ln-providers.mdx) and [Configuration](../configuration/index.mdx) pages for setup.
:::

## Managing accounts as an admin

Administrative account routes live under `/v1/accounts`:

- `POST /v1/accounts` creates or returns the account for a provider and subject.
- `GET /v1/accounts` and `GET /v1/accounts/{id}` list or fetch account aggregates.
- `PUT /v1/accounts/{id}` replaces the editable profile fields.
- `PUT /v1/accounts/{id}/permissions` replaces permissions for a local JWT account.
- `DELETE /v1/accounts/{id}` removes the account and its wallets, API keys, identities, preferences, and owned address data. The authenticated account cannot delete itself.

Reading requires `read:account`; creation, updates, permission grants, and deletion require
`write:account`. OAuth2/OIDC permissions come from the current token and cannot be written through
the account API. Local JWT permissions are stored on the account and take effect on subsequent
requests.

Account aggregates include the same lightweight wallet records used by the dashboard account and
wallet selectors. Full wallet resources remain behind wallet-by-ID endpoints.

## Managing API keys

For automation, create API keys instead of using the all-powerful sign-in JWT. Every key can use
the ownership-scoped `/v1/me` surface for its account. Its `permissions` array grants only
additional administrative endpoints, and each key can be revoked independently.

- [`POST /v1/me/api-keys`](https://api.numeraire.tech/docs#tag/me/POST/v1/me/api-keys) — generate a key for your account.
- [`GET /v1/me/api-keys`](https://api.numeraire.tech/docs#tag/me/GET/v1/me/api-keys) and [`GET /v1/me/api-keys/{id}`](https://api.numeraire.tech/docs#tag/me/GET/v1/me/api-keys/{id}) — list and look up keys.
- [`DELETE /v1/me/api-keys`](https://api.numeraire.tech/docs#tag/me/DELETE/v1/me/api-keys) and [`DELETE /v1/me/api-keys/{id}`](https://api.numeraire.tech/docs#tag/me/DELETE/v1/me/api-keys/{id}) — revoke keys.

```bash
curl -X POST https://your-instance:3000/v1/me/api-keys \
  -H 'Authorization: Bearer <jwt>' \
  -H 'Content-Type: application/json' \
  -d '{
    "name": "payments-bot",
    "permissions": [],
    "expiry": 2592000
  }'
```

`name` and `permissions` are required; `description` and `expiry` (seconds) are optional. API keys
are account-scoped and can only receive administrative permissions that are a subset of the
creator's. An empty permission list still permits `/v1/me` account-wallet operations. Wallet
operations require the wallet ID in the request path. The full `key` value is returned **only once
on creation** — store it securely, as it cannot be retrieved again.

The available permission scopes are:

| Scope               | Additional administrative access                 |
| ------------------- | ------------------------------------------------- |
| `read:account`      | List and read account aggregates.                 |
| `write:account`     | Create, update, grant, and delete accounts.       |
| `read:wallet`       | Read wallets and balances across accounts.        |
| `write:wallet`      | Create and manage wallets across accounts.        |
| `read:transaction`  | List and read administrative invoices/payments.   |
| `write:transaction` | Generate invoices and send administrative payments. |
| `read:ln_address`   | Read administrative Lightning Address resources. |
| `write:ln_address`  | Manage administrative Lightning Address resources. |
| `read:ln_node`      | Read Lightning node information.                   |
| `write:ln_node`     | Perform Lightning node operations.                |
| `read:btc_address`  | Read on-chain Bitcoin addresses.                  |
| `write:btc_address` | Generate and delete on-chain Bitcoin addresses.   |
| `read:api_key`      | List and read API keys.                           |
| `write:api_key`     | Create and revoke API keys.                       |

Present the key in the `Api-Key` header on each request:

```bash
curl https://your-instance:3000/v1/me/wallets/<wallet-id>/payments \
  -H 'Api-Key: YOUR_API_KEY' \
  -H 'Content-Type: application/json'
```

:::tip
Always send request bodies as JSON with the `Content-Type: application/json` header. For more on API keys and building integrations, see the [API Keys](../../developers/api-keys.mdx) developer guide.
:::
